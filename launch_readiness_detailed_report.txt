4) CRITICAL FINDINGS (P0 - MUST FIX BEFORE PUBLIC LAUNCH)
====================================================================

P0-1) Non-production network base URL and cleartext transport
- Evidence:
  - lib/core/network/baseUrl.dart:4 -> http://192.168.0.24:3000
  - android/app/src/main/AndroidManifest.xml:22 -> android:usesCleartextTraffic="true"
  - android/app/src/main/AndroidManifest.xml:25 -> android:requestLegacyExternalStorage="true"
- Risk:
  - Production users cannot reach private LAN backend.
  - HTTP traffic can be intercepted.
  - Play Store/security reviewers can flag weak transport posture.
- Required action:
  - Move base URLs to environment/flavor config.
  - Require HTTPS for production.
  - Disable cleartext for release.

P0-2) Privacy-sensitive device data collection without explicit user control
- Evidence:
  - lib/features/onboarding/pages/device_info.dart:24, 28, 44, 91, 138
  - Called during onboarding/auth/preference flows:
    - lib/features/onboarding/pages/choos_fav_players_list.dart:53
    - lib/state/application/auth/auth_bloc.dart:95
    - lib/state/application/set_preference/set_preference_bloc.dart:86
- Risk:
  - Collecting deviceId + IP + geolocation + network + battery can trigger compliance exposure.
  - User trust impact if this is not clearly disclosed and optional.
- Required action:
  - Add explicit consent gate and policy disclosure before collection.
  - Minimize payload to strictly required fields.
  - Remove raw IP storage unless strictly necessary.

P0-3) Sensitive values logged in app runtime
- Evidence:
  - lib/Homepage.dart:91 -> logs FCM token value
  - lib/services/fcm_service.dart:443, 480 -> logs token details
  - lib/features/payment/pages/chapa_payment.dart:95, 141-153 -> logs payment internals
- Risk:
  - Sensitive data may leak through logs/crash reports/support dumps.
- Required action:
  - Introduce sanitized logger with release suppression.
  - Never print tokens, auth headers, or payment response bodies.

P0-4) State correctness risk in standings state model
- Evidence:
  - lib/state/bloc/standings/bloc/content_state.dart:100+ mutable fields
  - lib/state/bloc/standings/bloc/content_state.dart:184-189 props includes only subset
  - lib/state/bloc/standings/bloc/content_event.dart has mutable event fields
- Risk:
  - Equatable state equality can fail to detect meaningful changes.
  - UI may not rebuild reliably; stale state bugs become hard to reproduce.
- Required action:
  - Make state/event models immutable (final fields).
  - Correctly include relevant fields in props or split large state.
